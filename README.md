    - ограничения на типы:
        - заголовок обязателен (1 - 100 символов)
        - описание опционально, если есть 1-1000 символов
        - поле completed обязательно
        - created_at обязательно, менять нельзя
        - completed_at опционально
        - если completed == true, то completed_at IS NOT NULL и completed_at >= created_at
        - если completed == false, то completed_at IS NULL
3. Просмотр статистики по задачам (по всей системе/по конкретному пользователю/в определенный промежуток)
    - всего задач выполнено
    - всего задач создано
    - процент выполненных задач
    - среднее время выполнения задачи

### Архитектура сущностей и обеспечение конкурентного доступа

В рамках проектирования базы данных выделены две ключевые сущности: `User` и `Task`. Между ними реализована классическая связь «один ко многим» (One-to-Many): один пользователь может выступать автором множества задач, при этом каждая задача жестко привязана к конкретному автору через внешний ключ `author_user_id` в таблице `Tasks`. Первичным ключом (`id`) для обеих таблиц является автоинкрементный целочисленный тип `SERIAL`.

Для предотвращения состояний гонки (Race Conditions) и эффекта утери обновлений (Lost Update) при параллельных запросах, в обеих сущностях задействован механизм оптимистичной блокировки (Optimistic Concurrency Control).

**Алгоритм работы механизма:**

1. В таблицы `Tasks` и `Users` интегрировано дополнительное целочисленное поле `version`.
2. При выборке (чтении) данных из БД бэкенд всегда запрашивает текущее состояние сущности вместе с её актуальным номером версии.
3. На этапе модификации (обновления) данных в SQL-запрос закладывается проверка на соответствие исходной версии, а само поле версии инкрементируется в рамках того же шага.

Если в промежутке между чтением и записью параллельный процесс успеет изменить строку и обновить её версию, текущий запрос не затронет ни одной записи. В этом случае бэкенд фиксирует конфликт конкурентного доступа и генерирует соответствующую ошибку.

Выбор оптимистичной блокировки вместо пессимистичных локов обусловлен характером нагрузки на систему: одновременное обращение нескольких процессов к одной и той же строке данных потенциально является редким событием в полученных отношениях. Это позволяет избежать избыточных блокировок на уровне СУБД и сохраняет высокую производительность приложения при операциях чтения.

### Схема REST API

**Фичи пользователей:**

| Функциональность                            | API                                      |
| :------------------------------------------ | ---------------------------------------- |
| 1. Создавать нового пользователя            | POST /users                              |
| 2. Получать список пользователей            | GET /users?limit={limit}&offset={offset} |
| 3. Получать конкретного пользователя        | GET /users/{id}                          |
| 4. Менять данные существующего пользователя | PATCH /users/{id}                        |
| 5. Удалять существующего пользователя       | DELETE /users/{id}                       |
**Фичи задач:**

| Функциональность                     | API                                                        |
| ------------------------------------ | ---------------------------------------------------------- |
| 1. Создавать новую задачу            | POST /tasks                                                |
| 2. Получать список задач             | GET /tasks?user_id={user_id}&limit={limit}&offset={offset} |
| 3. Получать конкретную задачу        | GET /tasks/{id}                                            |
| 4. Менять данные существующей задачи | PATCH /tasks/{id}                                          |
| 5. Удалять существующую задачу       | DELETE /tasks/{id}                                         |
**Фичи статистики:**

| Функциональность                  | API                                                   |
| --------------------------------- | ----------------------------------------------------- |
| 1. Получить статистику по задачам | GET /statistics?user_id={user_id}&from={from}&to={to} |
